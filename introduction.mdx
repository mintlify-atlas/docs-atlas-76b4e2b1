## Overview

The Weather Microbatch ETL Pipeline is an end-to-end **production-style microbatch ETL system** that demonstrates real-world data engineering patterns. It extracts weather data from the OpenWeather API, transforms it into analytics-ready format, and loads it into a PostgreSQL data warehouse using an idempotent, retry-safe design.

The pipeline is built with Python and PostgreSQL, designed for 15-minute microbatch intervals, and structured to support orchestration with Apache Airflow and containerization with Docker.

### Extract layer

Fetches weather data from the OpenWeather API with:
- 5-second timeout per request
- Up to 3 retry attempts with exponential backoff (2s min, 30s max)
- Graceful error handling and logging

See `extract.py:9-20`

### Transform layer

Cleans and normalizes raw API data:
- Converts temperatures from Kelvin to Celsius
- Parses Unix timestamps to UTC datetime objects
- Structures data for database schema compatibility
- Uses pandas DataFrame for efficient processing

See `transform.py:6-25`

### Load layer

Persists data to PostgreSQL with:
- Foreign key resolution from dimension table
- Idempotent insert with `ON CONFLICT (city_id, recorded_at) DO NOTHING`
- Batch loading using `executemany` for performance
- Transaction safety with automatic rollback on errors

See `load.py:8-46`

## Data model

The pipeline uses a star schema with one dimension table and one fact table:

### City dimension table

| Column | Type | Description |
|--------|------|-------------|
| `city_id` | `BIGINT` | Primary key |
| `city_name` | `VARCHAR` | Unique city name |
| `latitude` | `DOUBLE PRECISION` | City latitude |
| `longitude` | `DOUBLE PRECISION` | City longitude |

**Constraints:** Unique constraint on `city_name`

### Weather fact table

| Column | Type | Description |
|--------|------|-------------|
| `weather_id` | `BIGINT` | Primary key |
| `city_id` | `BIGINT` | Foreign key to `city.city_id` |
| `weather_description` | `VARCHAR` | Weather condition (e.g., "Clear", "Rain") |
| `temperature_c` | `DOUBLE PRECISION` | Temperature in Celsius |
| `temperature_feels_like_c` | `DOUBLE PRECISION` | Feels-like temperature in Celsius |
| `humidity_percentage` | `INTEGER` | Humidity percentage |
| `pressure_hpa` | `INTEGER` | Atmospheric pressure in hPa |
| `wind_speed` | `DOUBLE PRECISION` | Wind speed in m/s |
| `recorded_at` | `TIMESTAMPTZ` | When the weather was recorded (event time) |
| `ingested_at` | `TIMESTAMPTZ` | When the data was loaded (defaults to current timestamp) |

**Constraints:** 
- Unique constraint on `(city_id, recorded_at)` ensures idempotency
- Foreign key relationship enforces referential integrity

<Note>
The `UNIQUE (city_id, recorded_at)` constraint is critical for idempotency. If the same weather reading is inserted multiple times, the `ON CONFLICT DO NOTHING` clause prevents duplicates.
</Note>
