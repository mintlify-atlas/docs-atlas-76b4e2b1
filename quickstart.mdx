## Installation

<Steps>
  <Step title="Clone the repository">
    Clone the Weather Microbatch ETL Pipeline repository to your local machine:

    ```bash
    git clone https://github.com/yourusername/weather-microbatch-etl-pipeline.git
    cd weather-microbatch-etl-pipeline
    ```
  </Step>

  <Step title="Install dependencies">
    Install all required Python packages using pip:

    ```bash
    pip install -r requirements.txt
    ```

    <Accordion title="View dependencies">
      The pipeline uses the following key dependencies:

      - **requests** (2.32.5) - HTTP library for API calls
      - **pandas** (3.0.0) - Data manipulation and transformation
      - **psycopg** (3.3.3) - PostgreSQL database adapter
      - **tenacity** (9.1.4) - Retry logic with exponential backoff
      - **python-dotenv** (1.2.1) - Environment variable management
      - **numpy** (2.4.2) - Numerical operations support
    </Accordion>
  </Step>

  <Step title="Set up the database">
    Create the required database schema in your PostgreSQL instance:

    <CodeGroup>
      ```sql City dimension table
      CREATE TABLE city (
          city_id BIGSERIAL PRIMARY KEY,
          city_name VARCHAR(100) UNIQUE NOT NULL,
          latitude DOUBLE PRECISION NOT NULL,
          longitude DOUBLE PRECISION NOT NULL
      );
      ```

      ```sql Weather fact table
      CREATE TABLE weather (
          weather_id BIGSERIAL PRIMARY KEY,
          city_id BIGINT NOT NULL,
          weather_description VARCHAR(100),
          temperature_c DOUBLE PRECISION,
          temperature_feels_like_c DOUBLE PRECISION,
          humidity_percentage INTEGER,
          pressure_hpa INTEGER,
          wind_speed DOUBLE PRECISION,
          recorded_at TIMESTAMPTZ NOT NULL,
          ingested_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
          FOREIGN KEY (city_id) REFERENCES city(city_id),
          UNIQUE (city_id, recorded_at)
      );
      ```
    </CodeGroup>

    <Note>
      The `UNIQUE (city_id, recorded_at)` constraint is essential for idempotent loading. It ensures that the same weather reading cannot be inserted twice.
    </Note>
  </Step>

  <Step title="Create logs directory">
    Create a directory for log files:

    ```bash
    mkdir -p logs
    ```

    The pipeline writes detailed execution logs to `logs/etl.log`.
  </Step>
</Steps>

## Configuration

<Steps>
  <Step title="Create environment file">
    Create a `.env` file in the project root directory:

    ```bash
    touch .env
    ```
  </Step>

  <Step title="Add configuration variables">
    Add the following environment variables to your `.env` file:

    ```bash .env
    # OpenWeather API Configuration
    API_KEY=your_openweather_api_key_here

    # PostgreSQL Database Configuration
    HOST_NAME=localhost
    DBNAME=weather_db
    PORT=5432
    USER=your_db_username
    PASSWORD=your_db_password
    ```

    <Warning>
      Never commit your `.env` file to version control. It contains sensitive credentials. The file is already included in `.gitignore`.
    </Warning>
  </Step>

  <Step title="Customize city list (optional)">
    By default, the pipeline tracks weather for these cities:

    ```python src/config.py
    cities = ["Chennai", "Bangalore", "Hyderabad", "Mumbai", "Delhi"]
    ```

    You can modify this list in `src/config.py` to track different cities:

    ```python src/config.py
    cities = ["New York", "London", "Tokyo", "Paris", "Sydney"]
    ```

    <Note>
      City names must match the OpenWeather API format. Use official city names for best results.
    </Note>
  </Step>
</Steps>

## Running the pipeline

<Steps>
  <Step title="Load city dimension data (first-time setup)">
    Before running the ETL pipeline for the first time, populate the city dimension table:

    ```python
    from src.extract import fetch_weather_data
    from src.load_cities import load_cities_data
    from src.config import cities

    # Fetch city coordinates from API
    cities_data = fetch_weather_data(cities=cities)
    
    # Load into dimension table
    load_cities_data(cities_data=cities_data)
    ```

    This step only needs to be run once, or when you add new cities to your configuration.
  </Step>

  <Step title="Run the ETL pipeline">
    Execute the main pipeline:

    ```bash
    python src/main.py
    ```

    You should see log output similar to:

    ```

    2026-03-03 10:15:25,457 | transform | INFO | Weather data transformation started
    2026-03-03 10:15:25,567 | transform | INFO | Weather data transformation finished successfully
    2026-03-03 10:15:25,568 | load | INFO | Weather data loading started
    ```

  </Step>

  <Step title="Verify the data">
    Check that weather data was loaded successfully:

    ```sql
    SELECT 
        c.city_name,
        w.weather_description,
        w.temperature_c,
        w.humidity_percentage,
        w.recorded_at,
        w.ingested_at
    FROM weather w
    JOIN city c ON w.city_id = c.city_id
    ORDER BY w.recorded_at DESC
    LIMIT 10;
    ```

    You should see weather records for all configured cities.
  </Step>
</Steps>

## Understanding the pipeline flow

The pipeline executes in three sequential stages:

<CodeGroup>
  ```python main.py
  import logging
  from extract import fetch_weather_data
  from transform import transform_weather_data
  from load import load_weather_data
  from config import cities
  from logger_config import setup_logger

  setup_logger()
  logger = logging.getLogger(__name__)

  logger.info("ETL pipeline started")

  # Stage 1: Extract
  weather_data = fetch_weather_data(cities=cities)
  
  # Stage 2: Transform
  cleaned_data = transform_weather_data(weather_data=weather_data)
  
  # Stage 3: Load
  load_weather_data(cleaned_data=cleaned_data)

  logger.info("ETL pipeline finished")
  ```

  ```python Extract example
  @retry(stop=stop_after_attempt(3), 
         wait=wait_exponential(multiplier=2, min=2, max=30))
  def fetch_city_weather_data(city):
      url = f"https://api.openweathermap.org/data/2.5/weather?q={city}&appid={API_KEY}"
      
      try:
          response = requests.get(url, timeout=5)
          response.raise_for_status()
          return response.json()
      except Exception:
          logger.debug(f"Retry weather data extraction for city {city}")
          raise
  ```

  ```python Transform example
  records.append({
      'city': data['name'],
      'weather_description': data['weather'][0]['main'],
      'temp_c': round((data['main']['temp']) - 273.15, 2),
      'temp_feels_like_c': round((data['main']['feels_like']) - 273.15, 2),
      'humidity_percentage': data['main']['humidity'],
      'pressure_hpa': data['main']['pressure'],
      'wind_speed': round(data['wind']['speed'], 2),
      'recorded_at': pd.to_datetime(data['dt'], utc=True, unit='s')
  })
  ```

  ```python Load example
  insert_query = """
  INSERT INTO weather(
      city_id, 
      weather_description, 
      temperature_c, 
      temperature_feels_like_c, 
      humidity_percentage, 
      pressure_hpa, 
      wind_speed, 
      recorded_at
  )
  VALUES(%s, %s, %s, %s, %s, %s, %s, %s) 
  ON CONFLICT (city_id, recorded_at) DO NOTHING;
  """
  cur.executemany(insert_query, cleaned_data)
  ```
</CodeGroup>

## Scheduling with cron

To run the pipeline as a microbatch job every 15 minutes, add a cron job:

```bash
# Edit crontab
crontab -e

# Add this line to run every 15 minutes
*/15 * * * * cd /path/to/weather-microbatch-etl-pipeline && /usr/bin/python3 src/main.py >> logs/cron.log 2>&1
```

<Tip>
For production deployments, consider using Apache Airflow or other orchestration tools for more sophisticated scheduling, monitoring, and error handling.
</Tip>

## Troubleshooting

<AccordionGroup>
  <Accordion title="API key authentication error">
    **Error:** `401 Unauthorized` when calling OpenWeather API

    **Solution:** 
    - Verify your API key is correct in the `.env` file
    - Check that your API key is activated (can take a few hours after signup)
    - Ensure you're using the correct API endpoint for your subscription tier
  </Accordion>

  <Accordion title="Database connection failed">
    **Error:** `psycopg.OperationalError: connection failed`

    **Solution:**
    - Verify PostgreSQL is running: `pg_isready`
    - Check database credentials in `.env` file
    - Ensure the database exists: `CREATE DATABASE weather_db;`
    - Verify network connectivity if using remote database
    - Check firewall rules for PostgreSQL port (default 5432)
  </Accordion>

  <Accordion title="Duplicate key error">
    **Error:** Even though you have `ON CONFLICT DO NOTHING`

    **Solution:**
    - This shouldn't happen with proper configuration
    - Verify the unique constraint exists: `\d weather` in psql
    - Check that `recorded_at` timestamps are being parsed correctly
    - Ensure `city_id` foreign keys are resolving properly
  </Accordion>

  <Accordion title="Retry exhausted errors">
    **Error:** All 3 retry attempts failed for API or database

    **Solution:**
    - Check internet connectivity for API calls
    - Verify API rate limits haven't been exceeded
    - For database retries, check connection pool settings
    - Review logs in `logs/etl.log` for specific error details
  </Accordion>

  <Accordion title="Temperature conversion looks wrong">
    **Issue:** Temperatures seem incorrect

    **Solution:**
    - OpenWeather API returns temperatures in Kelvin by default
    - The pipeline converts to Celsius: `(temp_kelvin - 273.15)`
    - To convert to Fahrenheit: `(temp_c * 9/5) + 32`
    - Verify the conversion in `src/transform.py:15-16`
  </Accordion>
</AccordionGroup>
