The load component inserts transformed weather data into a PostgreSQL database with automatic foreign key resolution, conflict handling, and retry logic.

## Overview

The `load.py` module provides a single function that:

- Connects to PostgreSQL database
- Resolves city names to foreign key IDs
- Handles duplicate records with conflict resolution
- Implements retry logic for transient failures
- Performs batch inserts for efficiency

## Function signature

```python load.py:8-46
@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=1, max=10))
def load_weather_data(cleaned_data):
    logger.info("Weather data loading started")

    try:
        with psycopg.connect(host=host, dbname=dbname, port=port, user=user, password=password) as conn:
            with conn.cursor() as cur:
                select_query = "SELECT city_name, city_id FROM city;"
                cur.execute(select_query)
                
                rows = cur.fetchall()
                city_id = {city_name: city_id for city_name, city_id in rows}
                
                cleaned_data['city_id'] = cleaned_data['city'].map(city_id)
                cleaned_data = cleaned_data.drop(columns = ['city'])
                cleaned_data = cleaned_data[[
                    'city_id',
                    'weather_description',
                    'temp_c',
                    'temp_feels_like_c',
                    'humidity_percentage',
                    'pressure_hpa',
                    'wind_speed',
                    'recorded_at'
                ]]
                cleaned_data = list(cleaned_data.itertuples(index = False, name = None))

                insert_query = """
                INSERT INTO weather(city_id, weather_description, temperature_c, temperature_feels_like_c, humidity_percentage, pressure_hpa, wind_speed, recorded_at)
                VALUES(%s, %s, %s, %s, %s, %s, %s, %s) ON CONFLICT (city_id, recorded_at) DO NOTHING;
                """
                cur.executemany(insert_query, cleaned_data)

                logger.info(f"Inserted {len(cleaned_data)} rows into database")
    except Exception as e:
        logger.exception(e)
        raise e
    
    logger.info("Weather data loading finished successfully")
```

## Foreign key resolution

The load process resolves city names to foreign key IDs from the `city` table:

<Steps>
  <Step title="Query city lookup table">
    ```python load.py:15-16
    select_query = "SELECT city_name, city_id FROM city;"
    cur.execute(select_query)
    ```
    Retrieves all city names and their corresponding IDs from the database.
  </Step>
  
  <Step title="Create mapping dictionary">
    ```python load.py:18-19
    rows = cur.fetchall()
    city_id = {city_name: city_id for city_name, city_id in rows}
    ```
    Builds an in-memory dictionary for efficient lookups: `{"London": 1, "New York": 2, ...}`
  </Step>
  
  <Step title="Map city names to IDs">
    ```python load.py:21
    cleaned_data['city_id'] = cleaned_data['city'].map(city_id)
    ```
    Uses pandas `map()` to replace city names with foreign key IDs.
  </Step>
  
  <Step title="Remove city name column">
    ```python load.py:22
    cleaned_data = cleaned_data.drop(columns = ['city'])
    ```
    Drops the city name column as it's no longer needed.
  </Step>
</Steps>

<Note>
The city table must be pre-populated with all cities before the ETL pipeline runs. Cities not found in the lookup table will have NULL city_id values.
</Note>


### Conflict handling

<Warning>
The `ON CONFLICT (city_id, recorded_at) DO NOTHING` clause prevents duplicate entries. If a record with the same city and timestamp already exists, the new record is silently ignored.
</Warning>

This conflict resolution strategy:
- Assumes a composite unique constraint on `(city_id, recorded_at)`
- Prevents duplicate weather readings for the same city at the same time
- Allows the pipeline to be safely re-run without creating duplicates
- Does not update existing records (idempotent inserts only)

## Retry logic

The load function implements automatic retry with exponential backoff:

```python load.py:8
@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=1, max=10))
```

### Retry configuration

| Parameter | Value | Description |
|-----------|-------|-------------|
| `stop` | `stop_after_attempt(3)` | Maximum 3 attempts |
| `wait` | `wait_exponential(multiplier=1, min=1, max=10)` | Exponential backoff |
| `min` wait | 1 second | Minimum wait between retries |
| `max` wait | 10 seconds | Maximum wait between retries |

### Retry timeline

<Steps>
  <Step title="Initial attempt">
    First database connection and insertion attempt.
  </Step>
  
  <Step title="First retry (if needed)">
    After 1 second, second attempt is made.
  </Step>
  
  <Step title="Second retry (if needed)">
    After 2 seconds, third and final attempt is made.
  </Step>
  
  <Step title="Failure">
    If all attempts fail, the exception is logged and re-raised.
  </Step>
</Steps>

## Database schema requirements

The load function assumes the following database schema:

### city table

```sql
CREATE TABLE city (
    city_id SERIAL PRIMARY KEY,
    city_name VARCHAR(100) UNIQUE NOT NULL
);
```

### weather table

```sql
CREATE TABLE weather (
    weather_id SERIAL PRIMARY KEY,
    city_id INTEGER REFERENCES city(city_id),
    weather_description VARCHAR(50),
    temperature_c NUMERIC(5,2),
    temperature_feels_like_c NUMERIC(5,2),
    humidity_percentage INTEGER,
    pressure_hpa INTEGER,
    wind_speed NUMERIC(5,2),
    recorded_at TIMESTAMP WITH TIME ZONE,
    UNIQUE(city_id, recorded_at)
);
```

<Warning>
The `UNIQUE(city_id, recorded_at)` constraint is required for the `ON CONFLICT` clause to work correctly.
</Warning>
